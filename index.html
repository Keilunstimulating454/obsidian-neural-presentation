About this presentation

OBSIDIAN Neural — Workshop

This presentation documents a practical experiment: building an open-source VST3/AU plugin that integrates AI music generation into a live performance workflow.

It covers the technical architecture, the design decisions, the results — and the broader questions this work raises about how AI models could reshape the music industry's economic model.

Anthony Charretier is an independent developer and musician based in Grenoble, France. He performs as InnerMost47 and builds open-source tools at the intersection of AI and music.

obsidian-neural.com · github.com/innermost47

Context

Diffusion models in a DAW existed. None were built around the performance workflow.

By 2024, diffusion-based audio generation models had matured — Stable Audio, MusicGen, AudioLDM. Some had even been integrated into DAWs as VST plugins (TextToSample, MusicGen local). But these were single-track tools with no BPM sync, no MIDI control, no multi-track sampler. The workflow was always: generate → export → import. OBSIDIAN Neural removes that friction — audio is time-stretched server-side via RubberBand to match the host BPM, then loaded directly into one of 8 sampler tracks. Whether you're on stage or in the studio, the loop is ready to play.

Live performance has fundamentally different constraints:

  • Generation must be non-blocking — you can't pause a set
  • Output must integrate with DAW tempo and structure
  • The interface must be operable under performance conditions (dim light, stress, hardware controllers)
  • Latency is acceptable if it's predictable and the workflow absorbs it

No existing tool addressed this combination of constraints. The hypothesis: a sampler-based approach, where AI generates loops asynchronously while the performer plays previously loaded material, could make this work.

Technical Architecture

VST → Central Server → GPU Network

OBSIDIAN Neural is built as a distributed system with three layers:

  • VST3/AU Plugin (client) — runs inside the DAW (Bitwig, Ableton, Logic…). Sends text prompts or drawing data, receives WAV files. Handles BPM sync, track management, MIDI triggering. Each track has 4 pages, each page has its own model selector — up to 256 different model/page combinations per session.

    • Each page also includes:

    • Step sequencer — 16 steps × up to 4 measures, with 8 independent sequences per page. Lets you trigger specific sections of a loop at precise moments.
    • Waveform visualizer — with start/end loop markers and one-shot markers to isolate exactly the portion of audio you want to loop or trigger.

    At the track level (not per page):

    • Beat repeat — per track, for live stutter and rhythmic variation effects.
    • Mobile controller app (Flutter, open source) — controls all 8 tracks via USB MIDI with bidirectional feedback. No Bluetooth, no latency.
  • Central Server (FastAPI) — authenticates requests, routes jobs to available provider nodes, manages the credit system, and falls back to fal.ai (Stable Audio Open) when no node is online.
  • Provider Nodes (community GPU) — Docker containers running on contributors' GPUs. Each node runs two things: a Gemma4 LLM via Ollama (prompt optimization and draw-to-audio interpretation), and 8 specialized audio models. If no node is available, both layers fall back to fal.ai — LLM inference and Stable Audio Open for audio. Providers earn 85% of net subscription revenue, split equally.

Generation takes approximately 30 seconds. The plugin's 8-track sampler structure is designed to absorb this latency: while the performer plays 3–4 loaded tracks, new material is generating in the background.

The LLM layer (Gemini 2.5 Flash) optimizes prompts before sending them to the audio models, and interprets visual drawings for the draw-to-audio feature.

AI Models

8 Specialized Engines, One Per Track

Per page: 16-step sequencer × 4 measures × 8 sequences  ·  Waveform editor with loop markers  ·  Per track: beat repeat  ·  Mobile controller app

Each of the 8 tracks has a model selector — and each of the 4 pages per track has its own setting. That means track 1 page A can run Foundation-1, while track 1 page B runs StableBeaT. Nothing is fixed. You can also run all 8 tracks on the same model if that's what you need. The 8 available models, all fine-tuned from Stable Audio Open by independent creators:

  • Stable Audio Open 1.0 — versatile full-mix textures, 80–160 BPM
  • Foundation-1 (RoyalCities) — melodic & harmonic phrasing, 100–150 BPM
  • Audialab EDM Elements — leads, supersaws, plucks, ideal at 128 BPM
  • RC Infinite Pianos (RoyalCities) — grand & electric piano, optimal at 120 BPM
  • Vocal Textures Main (RoyalCities) — choral, operatic, atmospheric pads
  • SAO Instrumental Finetune — trap, lofi jazz, indie stems
  • StableBeaT — trap beats & 808 grooves, tight at 140 BPM
  • Gluten v1 — melodic trap & wavy motifs, optimal at 135 BPM

If no provider node is available, the system automatically falls back to Stable Audio Open via fal.ai. The performer never gets silence — just occasionally something unexpected.

This multi-model architecture also demonstrates something broader: that community-created fine-tunes can be integrated into a live performance tool without requiring the performer to understand the underlying model.

Experimental Feature

Draw-to-Audio: Sketching Sound

Each track includes a 512×512 drawing canvas. The performer sketches patterns, shapes, or abstract visuals using pencil, brush, or spray tools in 10 colors.

A vision LLM analyzes the drawing and generates a musical prompt based on visual elements. On provider nodes, this runs via Gemma4:e2b through Ollama — the same model used for text prompt optimization. If no node is available, it falls back to fal.ai.

  • Colors → Frequencies: black = bass, red = high energy, blue = ethereal, yellow = bright
  • Patterns → Rhythm: vertical lines = staccato, horizontal flow = sustained, circular = arpeggios, chaos = glitch
  • Brush type → Texture: pencil = sharp, brush = smooth, spray = diffuse

The original concept was developed by A.D. The drawing state, selected keywords, and brush settings are saved per track and per page (A/B/C/D) in the project.

This is intentionally experimental. The results are unpredictable. That's part of the point.

Reflection

What This Experiment Taught Me

Beyond the technical results, building and releasing OBSIDIAN Neural surfaced a set of questions I hadn't anticipated:

  • Latency is a design problem, not a technical failure. 30 seconds is too slow for melodic improvisation but perfectly workable for loop-based performance — if the interface is designed around it.
  • The community will extend your tool in ways you didn't plan. The 8-model architecture wasn't the original design. It emerged from what creators were publishing independently.
  • Granularity matters for live use. The step sequencer, loop markers, and beat repeat weren't afterthoughts — they're what makes a 30-second AI-generated loop actually usable in a live set. You need to be able to isolate 4 bars out of 30 seconds, and retrigger them precisely.
  • Open source changes the conversation. Because the code is public, researchers and musicians engage differently. They ask how it works, not just what it does.
  • AI as instrument vs. AI as composer is a real distinction. Most tools position AI as a replacement for human creative decisions. OBSIDIAN Neural treats it as a sound source — the performer still makes all the musical choices.

These lessons point toward a broader question: if community-created AI models can be integrated into live performance tools, what does that imply for how the music industry should think about model creation, ownership, and compensation?

A Broader Proposition

Model Packs as the Next Sample Packs

The music industry has an established model for sample packs: studios record sounds, package them, and sell licenses. What if the same logic applied to AI models?

The proposed model:

  • Studios or labels fine-tune AI models on their catalog or on dedicated recording sessions — instead of (or alongside) producing sample packs
  • Musicians are hired and compensated to record training data in the studio — a new category of session work
  • The resulting models are licensed rather than sold outright — enabling ongoing royalty streams for contributors
  • Performers integrate these models into their tools (like OBSIDIAN Neural) and pay per use, per subscription, or per license

This isn't a utopian proposal — it's a direct extrapolation of what's already happening informally on HuggingFace. The 8 models in OBSIDIAN Neural's network were all created by independent contributors. The infrastructure to compensate them more formally already exists (Stripe Connect, per-job billing).

The question isn't whether AI models will be part of music production. They already are. The question is whether the people whose sounds and styles trained those models will have any economic relationship with what gets built on top of them.

Central Server · Post-processing

From raw WAV to DAW-ready audio

Once the provider node returns a WAV file, the central server runs a deterministic post-processing pipeline before forwarding to the VST:

  • ffmpeg validation — the WAV is validated structurally. Invalid files are rejected before any further processing.
  • BPM detection — Librosa analyzes the generated audio to detect its actual tempo. Some models (Stable Audio Open, StableBeaT, SAO Instrumental, Gluten v1) are "imprecise" and require detection; others (Foundation-1, etc.) return a snapped BPM in their headers.
  • RubberBand time-stretch — the audio is stretched server-side to match the DAW's host BPM. No tempo mismatch reaches the performer.
  • Trim — the audio is trimmed to the exact duration requested by the VST.
  • Resample — resampled to match the DAW's sample rate (detected from the VST request).
  • Response headers — the WAV is returned with metadata: detected BPM, key, duration, provider name, fallback flag, remaining credits.

This pipeline runs on the central server — not on the provider node and not on the client. It ensures that regardless of which model or provider generated the audio, the VST always receives a correctly formatted, tempo-synced WAV.

Provider Verification · Proof-of-Work

Mel Spectrogram Fingerprinting

The verification system answers a simple question: how do you trust a GPU node you don't control?

The fingerprinting process:

  • A random prompt + seed is sent to the provider with a fixed duration
  • The returned WAV is converted to a mel spectrogram (128 mel bands, 22050Hz, fmax 8000Hz)
  • The mean vector across time is extracted as a 128-dimensional fingerprint
  • Cosine similarity is computed against a trusted reference fingerprint
  • If similarity is below the threshold for N consecutive rounds → automatic ban

The trusted reference:

  • One provider node is designated as "trusted"
  • It generates reference samples which are stored in an encrypted sample bank
  • If the trusted node is busy or unavailable, a random sample is drawn from the bank

Canary tests:

  • At random intervals (1–6h), the server sends deliberately invalid requests to providers
  • Valid providers must return HTTP 422 for all invalid inputs
  • A provider that accepts an invalid request has modified its code — it is banned immediately
  • Canary payloads include: invalid actions, empty prompts, out-of-range seeds, malformed model names, oversized payloads, invalid history roles

This is an open problem. Mel fingerprinting + canary tests is one pragmatic approach for a small distributed network. More robust alternatives — cryptographic commitments, zero-knowledge proofs, federated audit chains — are worth exploring. The full implementation is open source and contributions are welcome.
